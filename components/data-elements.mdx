Elementos is a Plain Old Java Object (POJO) that represents individual encyclopedia entries in HuelvaPedia. It's designed to work seamlessly with Firebase Realtime Database for automatic serialization and deserialization.

## Overview

The Elementos class stores all information about a topic or subtopic, including text content, images, location data, contact information, and navigation metadata. Each instance can represent either a parent topic (with subtopics) or a final leaf node with actionable information.

## Fields

<ParamField path="nombre" type="String">
  The name or title of the encyclopedia entry. Used both for display and as a key for Firebase queries in Secundaria activity.
</ParamField>

<ParamField path="foto" type="String">
  URL of the image representing this element. Loaded using the Picasso library in the Adaptador.
</ParamField>

<ParamField path="descripcion" type="String">
  A brief description or summary of the encyclopedia entry. Displayed as body text in the RecyclerView items.
</ParamField>

<ParamField path="ubicacion1" type="String">
  The first part of the location URI (e.g., "https://www.google.com/maps/search/?api=1&query="). Can be null if no location is associated.
</ParamField>

<ParamField path="ubicacion2" type="String">
  The second part of the location URI, typically the encoded address or place name. Combined with ubicacion1 to form a complete maps URL. Can be null.
</ParamField>

<ParamField path="telefono" type="String">
  Phone number associated with this element. Used with ACTION_DIAL intent for calling. Can be null.
</ParamField>

<ParamField path="ultimo" type="Boolean">
  Flag indicating whether this is a final-level element (true) or a navigation element with subtopics (false). Controls the behavior in Adaptador.
</ParamField>

<ParamField path="enlace" type="String">
  External web link URL (e.g., Wikipedia, official website). Opened with ACTION_VIEW intent. Can be null.
</ParamField>

## Firebase Integration

The Elementos class is designed as a Firebase-compatible POJO:

### Deserialization from Firebase

<CodeGroup>

```java In MainActivity/Secundaria
reference.addValueEventListener(new ValueEventListener()
{
    @Override
    public void onDataChange(@NonNull DataSnapshot snapshot)
    {
        listaElementos.clear();
        for (DataSnapshot dataSnapshot : snapshot.getChildren())
        {
            Elementos elemento = dataSnapshot.getValue(Elementos.class);
            listaElementos.add(elemento);
        }
        adaptador = new Adaptador(MainActivity.this, listaElementos);
        recyclerView.setAdapter(adaptador);
    }
    // ...
});
```

</CodeGroup>

Firebase automatically maps database fields to class properties using the getter/setter methods.

## Data Model Patterns

### Navigation Elements

Elements that lead to subtopics have:

```java
ultimo = false
ubicacion1 = null
ubicacion2 = null
telefono = null
enlace = null
```

**Behavior**: Tapping navigates to Secundaria activity with subtopics.

### Action Elements (Final Level)

Elements with actionable information have:

```java
ultimo = true
// One or more of these can be populated:
ubicacion1 = "https://www.google.com/maps/search/?api=1&query="
ubicacion2 = "Plaza de las Monjas, Huelva"
telefono = "959123456"
enlace = "https://es.wikipedia.org/wiki/..."
```

**Behavior**: Buttons appear for map, phone, or web based on which fields are populated.

## Key Design Decisions

### Split Location Fields

The location is split into `ubicacion1` (base URL) and `ubicacion2` (query) to allow:
- Flexible URL construction
- Proper encoding of the query part
- Reusability of the base URL across multiple elements

### Boolean vs boolean

Uses `Boolean` (wrapper) instead of `boolean` (primitive) to allow null values, which is important for Firebase compatibility and optional fields.

### String-based URLs

All URLs (photo, location, web link) are stored as Strings rather than Uri objects for Firebase serialization simplicity.

## Best Practices

<Note>
When creating Elementos objects for Firebase:
1. Always provide `nombre`, `foto`, and `descripcion` (required fields)
2. Set `ultimo` to indicate navigation vs. action behavior
3. Populate `ubicacion1` and `ubicacion2` together (both or neither)
4. Use null for optional fields that don't apply
5. Ensure `foto` URLs are publicly accessible for Picasso to load
</Note>

## Related Components

- **Adaptador**: Consumes Elementos to render RecyclerView items
- **MainActivity**: Fetches Elementos from the "principal" Firebase node
- **Secundaria**: Fetches Elementos from dynamically named Firebase nodes
- **Firebase Realtime Database**: Stores Elementos as JSON objects
