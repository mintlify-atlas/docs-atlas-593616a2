## Overview

HuelvaPedia uses a single, flexible data model class called **`Elementos`** to represent all encyclopedia entries, from top-level categories to terminal items with detailed information.

## Elementos Class

The `Elementos` class is a POJO (Plain Old Java Object) designed to work seamlessly with Firebase Realtime Database automatic serialization/deserialization.

### Complete Class Structure

```java Elementos.java
package com.example.huelvapedia;

public class Elementos {
    private String nombre;
    private String foto;
    private String descripcion;
    private String ubicacion1;
    private String ubicacion2;
    private String telefono;
    private Boolean ultimo;
    private String enlace;

    public Elementos() { //Constructor vacio
    }

    public Elementos(String nombre, String foto, String descripcion, 
                     String ubicacion1, String ubicacion2, String telefono, 
                     Boolean ultimo, String enlace) {
        this.nombre = nombre;
        this.foto = foto;
        this.descripcion = descripcion;
        this.ubicacion1 = ubicacion1;
        this.ubicacion2 = ubicacion2;
        this.telefono = telefono;
        this.ultimo = ultimo;
        this.enlace = enlace;
    }

    // Getters and setters...
}
```

<Note>
The empty constructor is **required** by Firebase for automatic deserialization from JSON to Java objects.
</Note>

<AccordionGroup>

### Navigation Field

<Accordion title="ultimo (Boolean)" icon="flag">
**Purpose**: Determines if this is a terminal item or has subcategories

**Values**:
- `false`: Item has subcategories, clicking navigates to Secundaria
- `true`: Terminal item, shows action buttons

**Critical Logic**:

```java Adaptador.java:54
if (!elemento.getUltimo()) // Si no es el último, se navega a la Activity Secundaria
{
    holder.itemView.setOnClickListener(v -> {
        Intent intent = new Intent(v.getContext(), Secundaria.class);
        intent.putExtra("Nombre", elemento.getNombre());
        v.getContext().startActivity(intent);
    });
}
else // Es el último elemento, solo permite pulsar botones de acción
{
    // Show location, phone, web buttons
}
```

**Required**: Yes (determines app behavior)
</Accordion>

## Field Validation Logic

The adapter checks for null values to determine button visibility:

```java Adaptador.java:50
boolean tieneUbicacion = elemento.getUbicacion1() != null && elemento.getUbicacion2() != null;
boolean tieneTelefono = elemento.getTelefono() != null;
boolean tieneEnlace = elemento.getEnlace() != null;
```

<Warning>
If action fields are null, their corresponding buttons are hidden via `setVisibility(View.GONE)`.
</Warning>

## Data Flow

### From Firebase to UI

<Steps>
  <Step title="Firebase Query">
    Activity queries Firebase database reference

  </Step>

  </Step>
  
  <Step title="Adapter Binding">
    Adaptador binds Elementos fields to views
    

  </Step>
  
  <Step title="UI Display">
    RecyclerView displays the bound data
  </Step>
</Steps>

## Design Rationale

<CardGroup cols={2}>
  <Card title="Flexible Schema" icon="expand">
    Same class handles categories and detailed items using nullable fields
  </Card>
  
  <Card title="Firebase Compatible" icon="fire">
    Empty constructor and getters/setters enable automatic serialization
  </Card>
  
  <Card title="Minimal Complexity" icon="minimize">
    Single POJO reduces codebase complexity
  </Card>
  
  <Card title="Extensible" icon="plus">
    Easy to add new fields without breaking existing functionality
  </Card>
</CardGroup>

## Firebase Database Structure

The Elementos class maps to this Firebase structure:

```
firebase-root/
├── principal/
│   ├── item1/
│   │   ├── nombre: "Historia"
│   │   ├── foto: "url"
│   │   ├── descripcion: "..."
│   │   ├── ultimo: false
│   │   └── ...
│   └── item2/
├── historia/  (normalized from "Historia")
│   ├── subitem1/
│   └── subitem2/
└── ...
```
